# Reference regression tests

Small, fast regression tests for `compassV2_2.py`, driven by **config.yaml
variants** rather than CLI flags: each case is a full copy of the repo's
`config.yaml` with a handful of overrides (tiny lattice, few logged steps,
fixed seed), run once, and checked in as reference data. The pytest tests
re-run each case and compare the fresh output against that reference with a
numerical tolerance.

This is a **regression** suite, not a physics-validation suite: it answers
"did anything change the numbers?", not "are the numbers physically
correct?". The test cases deliberately use large timesteps for speed, which
is why generating/running them prints a `stability monitor flagged N steps`
warning from `compassV2_2.py` -- that's expected and harmless here, not a
test failure.

## Layout

```
tests/
  cases.py                 -- the 3 test cases: name + config.yaml overrides
  helpers.py                -- shared machinery (see below)
  generate_reference.py     -- offline script: (re)builds fixtures/ + reference/
  conftest.py               -- pytest sys.path bootstrap
  test_reference_runs.py    -- the actual pytest tests
  fixtures/
    config_<case>.yaml       -- GENERATED: full config.yaml + that case's overrides
  reference/
    <case>/data/*.csv         -- GENERATED: reference CSV time series
    <case>/meta/*.json        -- GENERATED: reference run metadata
    <case>/states/*.npz       -- GENERATED: reference initial/final lattice state
```

Anything marked GENERATED is produced by `generate_reference.py` and checked
into git as the frozen reference -- don't hand-edit it.

## The 3 cases (`tests/cases.py`)

| case | geometry | field_mode | grid | ~logged rows |
|---|---|---|---|---|
| `square_static` | square | static | 3x3 (K=9) | 15 |
| `triangular_hysteresis` | triangular | hysteresis (log-spaced) | 3x3 (K=9) | 12 |
| `honeycomb_demag` | honeycomb | demag_rot | 3x3 (K=7) | 17 |

Each overrides `numerics.compass_engine.time.dt_factor` (bigger step, so
`t_sim` covers the run in a handful of steps), sets a fixed
`run.compass_engine.seed`, and an explicit `run.compass_engine.tag` (so the
output filename is stable and doesn't depend on the seed).

## How this actually exercises "config.yaml"

`compassV2_2.py` (and every other script in this repo) reads `config.yaml`
once at import time via `sim_config.load_config()`. To let each test case
run against its *own* config.yaml variant in the same pytest process, two
small mechanisms make that possible:

- `sim_config.load_config()` resolves its path from the `COMPASS_CONFIG_PATH`
  environment variable when no explicit path is given (falling back to the
  repo's own `config.yaml` otherwise -- this is additive-only and doesn't
  change default behavior for normal use).
- `tests/helpers.py::load_fresh_compassV2_2()` loads an independent copy of
  the `compassV2_2` module (via `importlib.util.spec_from_file_location`)
  with that environment variable set to the case's fixture path, so its
  module-level `CFG` is bound to the variant, not to the repo's real
  `config.yaml`.

This means the tests are proving that swapping `config.yaml` genuinely
changes what the simulator does end-to-end -- not just that a CLI flag
works.

## Running the tests

```
pytest tests/
```

All 3 cases together run in well under a second.

## What's compared, and with what tolerance

`tests/helpers.py` provides three comparisons, all with `rtol=1e-9`,
`atol=1e-12` (`np.allclose`):

- `compare_csv`: every logged row/column of the time-series CSV. Integer/
  label columns (`step`, `branch`, `forc_index`, `flip_field`, `flip_angle`)
  are compared exactly; everything else (fields, order parameters,
  energies, angular velocities) is compared with tolerance.
- `compare_metadata_derived`: the `derived` block of the run's metadata JSON
  (`omega0`, `T0`, `dt_s`, `Q`, `B_ref_T`, `n_steps`, ...) plus an exact
  match on the `config` block. Timestamps, wall-clock `runtime_s`, and
  absolute paths are deliberately excluded -- those legitimately differ
  every run.
- `compare_npz_state`: the final lattice state's `xs`, `ys`, `theta`,
  `omega` arrays.

The simulation is deterministic for a fixed seed, so in practice every
value matches to the bit; the nonzero tolerance exists to stay robust
across NumPy versions / CPU SIMD paths rather than because any drift is
expected.

Lattice PNGs (`images/`) are generated by `run_simulation()` but are **not**
part of the reference data or the comparison -- matplotlib rendering isn't
guaranteed pixel-stable across versions/fonts, so they're discarded by
`generate_reference.py` right after generation.

## Regenerating the references

Regenerate when you deliberately change something that should affect these
cases' output (a physics/numerics change in `compassV2_2.py`, or a default
in `config.yaml`), or when `config.yaml`'s schema changes in a way that
requires `tests/cases.py`'s overrides to be updated:

```
python3 tests/generate_reference.py
```

This overwrites `tests/fixtures/config_<case>.yaml` and
`tests/reference/<case>/`. **Always inspect `git diff` under `tests/`
afterward** before committing -- a diff here means either:

- an intentional change (expected: the physics/config change you just made
  should show up as a numeric diff in the reference data), or
- an accidental regression (not expected: investigate before committing).

Do not regenerate references just because a test failed and you want it to
pass again -- that defeats the point of a regression test. Regenerate only
when you can explain *why* the numbers are supposed to be different now.
